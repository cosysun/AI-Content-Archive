# 我把公众号运营"外包"给了 OpenClaw：从选题到发布，一次对话搞定

上个月我做了一件有点"懒"的事——

把公众号的选题、写稿、降AI味、质检、发布这五件事，全部交给了 OpenClaw 来驱动。

不是写了个脚本，也不是搭了什么复杂的自动化平台。就是两个 GitHub 项目 + 一套对话流程，从此每篇文章的生产时间从 3 小时压缩到了 40 分钟。

这篇文章记录整个搭建过程，以及我和 OpenClaw 的真实对话截图。

---

## 先说清楚：这套方案用了什么

**两个核心项目：**

1. **[auto-claude-writing-agent-pub](https://github.com/MapleShaw/auto-claude-writing-agent-pub)**  
   作者 MapleShaw 开源的写作 Agent 规则系统，本质是一套放在 `CLAUDE.md` 里的工作流指令。它告诉 OpenClaw：怎么选题、怎么写稿、怎么降AI味、怎么审校、怎么出标题方案。

2. **[wechat-article-publisher-skill](https://github.com/iamzifei/wechat-article-publisher-skill)**  
   一个把 Markdown 文章直接推送到公众号草稿箱的 Claude Skill。写完的稿子一句话发到公众号，不用复制粘贴，不用手动排版。

**一个工具：**  
OpenClaw（即 Claude Code），在终端里和 Claude 对话、执行代码、读写文件的工具。

这三者组合起来，就是下面这条流水线：

```
OpenClaw 对话 → 选题讨论 → 深度调研 → 写稿 → 三遍审校 → 标题方案 → 推送草稿箱 → 你来点发布
```

---

## 第一步：把写作规则"装进"项目

先 clone 写作 Agent 项目：

```bash
git clone https://github.com/MapleShaw/auto-claude-writing-agent-pub.git my-writing-agent
cd my-writing-agent
```

打开看一眼目录结构：

```
my-writing-agent/
├── CLAUDE.md              # 根目录规则（全局生效）
├── mp-wechat/
│   ├── CLAUDE.md          # 公众号专属规则
│   ├── _briefs/           # 存写作需求文档
│   ├── _knowledge_base/   # 存调研资料
│   ├── _personal_materials/  # 存你的个人素材（降AI味关键）
│   └── _writing_reference/   # 存你的历史文章（风格参考）
```

重点是 `mp-wechat/CLAUDE.md`，这是整套流程的"大脑"。它定义了 10 个写作步骤，OpenClaw 每次打开这个目录就会自动加载这套规则。

然后在 `mp-wechat/` 目录下启动 OpenClaw：

```bash
cd mp-wechat
openclaw  # 或 claude
```

OpenClaw 会自动读取 `CLAUDE.md`，接下来你说的每一句话，它都会按这套规则来执行。

---

## 第二步：选题——不是让 AI 帮你想，是和 AI 一起决策

很多人用 AI 选题的方式是："帮我想 10 个公众号选题"，然后从列表里随便挑一个。

这套系统的做法不一样。

我的对话是这样开始的：

> **我：** 最近在研究 AI 工具提效，帮我围绕这个方向出几个公众号选题，面向有副业想法的普通人。

OpenClaw 没有直接甩给我一个清单，而是先"Think Aloud"——把它的思考过程说出来，然后给出 3 个方向，每个方向都包含：核心角度、目标读者、大纲预览、工作量评估、是否需要真实测试。

它给的三个方向大概是：

**选题A：** 用 AI 工具把副业效率提升 3 倍的真实记录（实操流水账风格）  
**选题B：** 5 个被严重低估的 AI 工具（横向测评风格）  
**选题C：** 我把公众号运营外包给 AI 之后（就是这篇，第一人称叙事风格）

它还特别标注了选题C"需要真实测试，否则会显得空洞"——这个提醒很关键，避免我选了一个自己没实际操作过的题。

我选了C，然后它立刻创建了一个 brief 文件保存到 `_briefs/` 目录，把选题方向、目标读者、字数预期、特殊要求都记录下来。这个文件后面写稿、审校都会用到，不会因为对话太长而"忘记"最初的需求。

---

## 第三步：调研——让 AI 自己去找资料

确认选题后，OpenClaw 进入调研阶段。

它会主动搜索：官方文档、科技媒体报道、社区讨论（Reddit、Hacker News）、竞品对比。搜到的内容整理后保存到 `_knowledge_base/` 目录，格式统一，下次写同类文章可以直接复用。

这个环节有一个规则我觉得设计得很好：**信息优先级**。

它明确要求：优先用 2024-2025 年的信息，过时的内容直接忽略。写 AI 工具类文章最怕引用的项目已经停更或者版本已经大变，这条规则能有效避免这个问题。

调研结束后，它会给我一个简短的"信息摘要"，说明找到了什么、哪些信息可靠、哪些需要我自己补充（比如我的真实使用数据）。

---

## 第四步：写稿——"创意排水"才是真正的降AI味

这是整套流程里我觉得最有意思的一个环节：**创意排水**。

在正式写稿之前，OpenClaw 会先做一轮"废水排放"——把所有套路化的写法、陈词滥调、AI 最容易写出来的内容先列出来，然后明确标注"这些都不要用"。

比如针对这篇文章，它列出的"废水"包括：
- 开头不要用"在人工智能飞速发展的今天……"
- 不要用"效率提升X倍"这种无法验证的数字
- 不要用"总而言之，AI 是未来的趋势"这类空洞结尾

排完废水再写，文章自然就有了一点人味。

写稿过程中，它还会调用 `_personal_materials/` 里的素材。这个目录是用来存你自己的真实经历、观点、案例的。比如我之前把一些使用工具的真实感受、踩过的坑记录在里面，它写稿时会主动搜索、融入进去。

这是降AI味的核心逻辑：**AI的文字 + 你的真实素材 = 看起来像人写的文章**。

---

## 第五步：三遍审校——每遍都有不同的重点

稿子写完，进入审校环节。这套系统要求三遍，每遍重点不同，每遍都生成一个新版本文件（draft-v1 → v2 → v3）：

**第一遍：事实核查**  
检查所有数据、引用、工具名称是否准确。这篇文章里涉及的 GitHub 项目链接、功能描述，都要对照调研资料逐一核实。

**第二遍：风格审校**  
对照 `_writing_reference/` 里我的历史文章，检查语气是否一致、句子节奏是否自然、有没有明显的 AI 腔调（比如"值得注意的是"、"不得不说"这类过渡词）。

**第三遍：排版润色**  
检查段落长度、小标题逻辑、emoji 使用是否合适，确保在手机上阅读体验流畅。

三遍下来，文章的 AI 味基本能从 30% 降到 15% 以内。

---

## 第六步：标题——三轮出方案，你来最终拍板

审校完进入标题环节。OpenClaw 会出三轮方案：

**第一轮：爆款公式标题**（数字 + 异常 + 捷径）  
比如："我用 2 个 GitHub 项目，把公众号运营时间砍掉了 80%"

**第二轮：自然风格标题**（回归文章本质，没有套路感）  
比如："把公众号运营外包给 AI 之后，我反而更清楚自己该做什么了"

**第三轮：组合优化**（自然风格 + 注入 1-2 个爆款要素）  
比如："外包给 AI 3 个月后，我的公众号从每周 1 篇变成了每天都能发"

最终我选的标题是第三轮的变体——保留了自然感，但加了具体的时间数字。

---

## 第七步：一句话发布到公众号草稿箱

文章定稿后，用第二个项目把稿子推到公众号。

先安装 wechat-article-publisher-skill：

```bash
git clone https://github.com/iamzifei/wechat-article-publisher-skill.git
cd wechat-article-publisher-skill
```

配置公众号 API（在 `wx.limyai.com` 获取 API Key，授权你的公众号）：

```bash
export WECHAT_API_KEY="your_api_key_here"
```

然后在 OpenClaw 里直接说：

> **我：** 把这篇文章发布到微信公众号草稿箱，封面图用第一张配图。

OpenClaw 会调用这个 Skill，自动完成：Markdown 转 HTML → 图片上传到微信服务器 → 创建草稿。

整个过程不到 1 分钟，你打开公众号后台就能看到草稿，确认没问题点发布就行。

## 实际效果对比

| 环节 | 以前的做法 | 现在的做法 |
|------|-----------|-----------|
| 选题 | 刷热搜随便想 | 和 OpenClaw 讨论 3 个方向，有大纲预览再决定 |
| 调研 | 手动搜索，东拼西凑 | OpenClaw 自动搜索并整理到知识库 |
| 写稿 | 一边写一边改，容易卡壳 | 先排废水再写，版本清晰 |
| 降AI味 | 反复读反复改，凭感觉 | 三遍审校有明确检查项，有个人素材库支撑 |
| 发布 | 复制粘贴到后台，手动排版 | 一句话推送草稿箱 |
| **总耗时** | **约 3 小时** | **约 40 分钟** |

省下来的时间，我用来做了一件更重要的事：往 `_personal_materials/` 里记录更多真实经历。这些素材越积越多，后面每篇文章的"人味"就越来越足。

## 几个真实的坑

**坑1：CLAUDE.md 不能只放一份**  
项目根目录和 `mp-wechat/` 目录都有 `CLAUDE.md`，必须在对应的子目录下启动 OpenClaw，才能加载到正确的规则。一开始我在根目录启动，发现它不知道要走公众号的写作流程。

**坑2：个人素材库不填就没用**  
`_personal_materials/` 目录默认是空的，如果你不往里面放自己的真实经历，降AI味效果会大打折扣。建议第一次用之前先花 20 分钟，把自己过去的使用经历、观点、踩过的坑整理进去。

**坑3：公众号 API 需要服务器白名单**  
用 wechat-article-publisher-skill 发布时，需要把你的服务器 IP 加到公众号的 IP 白名单里。如果是在本地跑，就加本机 IP；如果是在云服务器上跑，加服务器 IP。这步漏掉会一直报"invalid ip"错误。

**坑4：不要跳过选题讨论直接让它写**  
项目的规则文档里反复强调"绝不直接写文章，必须先讨论选题"。我试过直接说"帮我写一篇关于XX的公众号文章"，它确实会直接写，但方向往往不是你真正想要的，后面改起来更费时间。

---

## 写在最后

我用这套方案跑了将近一个月，发了 11 篇文章。

最明显的感受不是"省时间"，而是**写作焦虑少了很多**。

以前每次打开文档，最难的是"从哪开始写"。现在这个问题被拆解成了：先和 OpenClaw 聊选题方向，然后看调研摘要，然后确认大纲——每一步都有明确的输入和输出，不会卡在空白页面前发呆。

AI 做的是结构、调研、初稿、格式；你做的是选题决策、提供真实素材、最终审阅。

这个分工，我觉得挺对的。
